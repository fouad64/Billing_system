## 🛠️ 2. Core Development Commands

### **A. Integrated Build & Deployment**
Use this command to build the frontend, package the application, and start the server:
```bash
./mvnw clean package cargo:run
```

### **B. Frontend Development Mode**
Execute for rapid UI development and styling with hot-reloading:
```bash
cd frontend && npm install && npm run dev
```
> [!IMPORTANT]
> **DEVELOPMENT ONLY**: This mode is strictly for UI/UX styling. It operates on a separate port (`5173`) and **cannot** be used for end-to-end testing, database verification, or production environments. For all functional testing involving the database and API logic, utilize the **Integrated Build (Section A)**.

### **C. CDR Engine Standalone Execution**
Verify the CDR Rating engine without full server deployment:
```bash
./mvnw exec:java -Dexec.mainClass="com.billing.cdr.CDRParser"
```

## 📡 3. API Endpoint Documentation

### **Customer Endpoints (`/api/customer/*`)**
- `GET /profile`: Returns profile information for the authenticated session.
- `GET /contracts`: Returns MSISDNs and associated plans for the session user.
- `GET /invoices`: Returns historical billing records for the session user.

### **Admin Endpoints (`/api/admin/*`)**
- `GET /cdr`: Returns paginated CDR records (50 per page).
- `POST /cdr`: Initiates an import and rating scan of the `input/` directory.
- `GET /contracts`: Retrieves all system contracts with customer resolution.
- `GET /bills`: Retrieves recent invoice records across the network.

## 🗄️ 4. Database Schema Overview
The system architecture utilizes 11 core tables for the billing lifecycle:
1. **`user_account`**: Centralized authentication and profile repository.
2. **`contract`**: Associates an MSISDN with a specific `rateplan` and tracks `available_credit`.
3. **`cdr`**: Repository for call records; utilizes a `rated` flag for processing status.
4. **`contract_consumption`**: Aggregates usage metrics for service bundles.
5. **`bill` / `invoice`**: Final financial records generated by the billing engine.

---

## 🛡️ 5. Security & Request Filtering
The `AppFilter.java` component intercepts all incoming requests to enforce security policies:
- **Exempt Routes**: `/api/public/*`, `/login`, `/register`.
- **Authenticated Routes**: Requires a valid `HttpSession`.
- **Session Enforcement**: Unauthorized requests receive a standard `401 Unauthorized` status or a redirect to the login interface.

## 🆘 6. Troubleshooting & Operational Procedures

### **UI Synchronization**
The application serves a compiled version of the frontend. To synchronize UI changes with the backend, execute `./mvnw package` to update the application archive.

### **CDR Import Execution**
The parser resolves the `input/` directory relative to the process execution path. Ensure commands are executed from the project root directory to maintain path integrity.

### **Database Connectivity**
Connection management is handled by HikariCP. Ensure the connection string in `DB.java` includes the required SSL parameters for cloud database environments.
